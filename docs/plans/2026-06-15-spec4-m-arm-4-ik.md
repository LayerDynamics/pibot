## Goal

Add Cartesian-pose → joint-angle solving behind the existing `JointSolver` seam, using numeric `ikpy`
loaded from the M-ARM-3 model — **without** pulling numpy/ikpy into the stdlib-light core at import
time. Then expose Cartesian jog/move through the M-ARM-1 control surface.

## Realizes

  intents).
- **Blocks:** M-ARM-5 Cartesian-linear paths.

## Tasks

### 4.1 — `arm-ik` extra + CI install

- **What:** add `arm-ik = ["ikpy>=3.3.1", "numpy>=1.22,<2"]` to `pyproject.toml`
  (`[project.optional-dependencies]`); the numpy `<2` pin keeps it co-installable with `[ml]`. Add it to
  `deploy/requirements.txt` (robot-only). Make `scripts/check.sh` / CI install `.[arm-ik]` so the IK/FK
  tests run **hermetically with zero skips** (numpy in the *test* env is fine — the boundary is about
  *core module import*, enforced by 4.3).
- **Files:** `pyproject.toml`, `pibot/deploy/requirements.txt` (or the deploy requirements source),
  `scripts/check.sh`.
- **Test-first:** a smoke test asserting `import ikpy` succeeds in the test environment.
- **Done when:** `.[arm-ik]` installs in CI and the smoke test passes.

### 4.2 — IKSolver behind the seam

- **What:** add `IKSolver` to `pibot/arm/kinematics.py` implementing the existing `JointSolver` Protocol
  — lazy-import ikpy, build the chain via `Chain.from_urdf_file` (M-ARM-3 model), `solve(pose) ->
  {jid:degrees}` with the rad→deg + joint-reorder shim, and a **reachability + joint-limit check**
  (out-of-reach / out-of-limit → raise a clear error; never emit unclamped joint targets).
- **Files:** `pibot/arm/kinematics.py`.
- **Test-first:** extend `tests/test_arm_kinematics.py` — **FK∘IK ≈ identity** within tolerance for
  reachable poses; an out-of-reach pose raises.
- **Done when:** IK round-trips reachable poses and rejects unreachable ones; gate green.

### 4.3 — Import-boundary guard

- **What:** prove the `[ml]`/stdlib-light invariant (NFR-2): after importing `pibot.arm`, `pibot.cli`,
  and `agent.app`, neither `numpy` nor `ikpy` is in `sys.modules`. Implement any lazy-import fixes the
  test surfaces.
- **Files:** `tests/test_arm_import_boundary.py` (new); fixes in `pibot/arm/*` as needed.
- **Test-first:** the boundary test itself (write it first; make the imports lazy until it passes).
- **Done when:** core/CLI/agent import numpy-free; test green.

### 4.4 — Cartesian jog / move

- **What:** extend `/arm/control` to accept a Cartesian pose/delta intent → `IKSolver.solve` →
  `ArmManager.move_synchronized`; thread through `AgentClient`/`RobotLink` (`arm_move_cartesian`),
  `POST /api/arm/move-cartesian`, `pibot arm move-xyz`, and an Arm-screen Cartesian control. The agent
  loads the `IKSolver` lazily and returns a clean error if `arm-ik`/model are absent.
- **Files:** `agent/app.py`, `pibot/control/client.py`, `pibot/mc/robot_link.py`,
  `pibot/mc/routes_arm.py`, `pibot/cli.py`, `app/src/screens/Arm.tsx`, `app/src/stores/armStore.ts`.
- **Test-first:** extend `tests/test_agent_arm.py` — Cartesian intent → joint targets (via a stub/loaded
  solver); unreachable → nak; no `arm-ik` → clean "IK unavailable" response.
- **Done when:** Cartesian move/jog works through the surface; gates green.

## Notes / risks

- **Open-loop accuracy (SPEC R4):** require homing before any Cartesian move; keep speeds conservative;
  document that pose accuracy is bounded by open-loop stepper repeatability.
- **Singularity / unreachable (SPEC R5):** the post-solve reachability + limit check is mandatory — a bad
  IK result must be rejected, never sent to the motors.
- numpy version: pin `<2` so `arm-ik` coexists with `[ml]` on the robot.
